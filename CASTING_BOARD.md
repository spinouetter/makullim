# 캐스트보드 이미지 분석 가이드 (에이전트용)

> **이 문서는 사용자가 아니라 작업하는 에이전트(Claude 또는 다른 에이전트)가 참조하는 절차서다.**
> 캐스트보드(공연장 로비의 회차별 캐스팅 안내판) **사진이 올라오면 이 순서대로 판독·대조·기록**한다.
>
> **문서 유지보수(중요)**: 작업 중 이 문서가 실제와 어긋났다고 느끼거나, 추가로 정리해둘 절차·주의점·판독 테크닉이 생기거나, 데이터 구조/규칙이 바뀐 것을 발견하면 — **에이전트 판단으로 이 문서를 그때그때 업데이트한다.** (문서를 최신으로 유지하는 것도 작업의 일부다.)

---

## 원칙: 데이터 주도

이 저장소는 **멀티 공연**이고 코드·문서는 **하드코딩을 피한다**. 배역 구성·그룹 구조·팀 명단은 공연마다 다르므로, **절차는 공연 불문으로 두고 구체값은 항상 그 공연의 데이터(`shows/<id>/*.json`)를 그 자리에서 읽어** 판단한다. 이 문서에 특정 공연의 배역명·배우명을 나열해 굳히지 않는다.

---

## 0. 한눈에 (요약 절차)

1. **대상 회차 결정** — 사용자가 일시를 주면 그걸, 안 주면 *미정 배역이 남은 회차 중 지금 가장 임박한(soonest) 회차*.
2. **어느 공연(show)인지 식별** — 보드 로고·배역 구성으로 show id 추정 + 현재 스케줄과 대조.
3. **배역별 배우 판독** — 이미지에서 `배역 → 배우`를 뽑고, 그 공연 `casts.json` 로스터로 이름 교정.
4. **기존 입력과 대조** — 대상 회차에 이미 값이 있으면 차이 확인(있으면 보고, 없으면 "일치"라고만).
5. **미정 채우기** — `schedule.json`에서 `[]`인 배역을 판독값으로 채운다.
6. **그룹 캐스트 팀 판정** — 아래 0/1/2+ 규칙.
7. **앙상블** — `casting_by_date.json`에만 기록, 변경·커버는 보고.
8. **두 파일 동시 갱신** — `schedule.json` + `casting_by_date.json`.

---

## 1. 대상 회차(공연 일시) 결정

- 사용자가 **일시를 지정하면 그 회차**를 대상으로 한다.
- **지정하지 않으면**: `schedule.json`에서 **`cast` 값이 `[]`(미정)인 배역이 하나라도 남아 있는 회차 중, 현재 시각에 가장 임박한(soonest, 오늘/가장 가까운 미래) 회차**를 대상으로 가정한다.
  - "가장 나중 날짜"가 아니라 **지금과 가장 가까운** 회차다.
  - 캐스트보드 사진은 보통 곧/방금 올라간 회차의 미확정 배역을 채우려 올라오기 때문이다.
- **대결(match, 승부)의 결과는 대상 회차 추측에 고려하지 않는다.** 승부 결과는 캐스팅과 무관하고 보드에도 없다.
- 대상이 애매하면(여러 회차가 후보) **사용자에게 확인**받는다.

> **미정의 표현**: 회차 `cast[배역]`이 **빈 배열 `[]`** 이면 미정. (문자열 `"미정"`은 쓰지 않는다.)

---

## 2. 어느 공연(show)의 캐스트보드인지 식별

- 보드 상단 로고·제목·배역 구성으로 어떤 공연인지 판별하고, **그 공연 폴더**(`shows/<id>/`)의 데이터를 기준으로 대조한다.
- 배우 이름은 작아 오독하기 쉽다. 읽은 이름은 **반드시 그 공연 `casts.json` 로스터의 이름과 매칭해 교정**한다.

### 공연별 캐스트 보드 차이
- 배역 수·그룹 구조·팀 유무·레이아웃은 **공연마다 다를 수 있다.**
- 특정 공연 보드의 특이점(예: 그룹/팀 구성, 배치 순서)은 **그 공연 보드를 실제로 다룰 때 이 섹션에 공연별로 정리**한다. 아직 확인 안 된 공연은 비워 둔다.

---

## 3. 데이터 구조 레퍼런스

| 파일 | 역할 | 캐스팅 표현 |
|---|---|---|
| `shows/<id>/casts.json` | **로스터 + 그룹 정의**(정답지) | 배역별 배우 목록, `role: cast\|cover\|standby`. 그룹은 `group:true` |
| `shows/<id>/schedule.json` | **회차별 메인 캐스팅** (앱 Schedule/통계) | `cast: { "배역": ["배우"] }`, **미정=`[]`**, 그룹 배역은 **팀/조합명** |
| `shows/<id>/casting_by_date.json` | **회차별 확장 캐스팅** (Finale) | `"YYYY-MM-DD HH:MM": { 배역: "배우", …, "앙상블":[[이름,역할]…] }` |
| `shows/<id>/matches.json` | 대결(승부) 정의 | 결과는 `schedule.json` 회차의 `match`에 저장 |

### `casts.json`
- **일반 배역**: `{ id, role, fullName?, shortName?, actors:[{name, role}] }`
  - `role` 값: `cast`(정캐) / `cover`(커버) / `standby`(스탠바이).
- **그룹 배역**(`group:true`): 여러 배우가 **팀/조합 단위**로 함께 움직이거나(팀형), 개별 멤버 명단만 있는(명단형) 배역.
  - 팀형: `actors`에 팀/조합 목록, `members`에 각 세부 배역. 세부 배역은 팀 공통이면 단일 `name`, 팀마다 다르면 `byTeam:{ "<팀>":"<배우>" }`.
  - 명단형: `members:[{ name, roles:[…] }]` (팀 없음).
  - **구체 팀/멤버는 그 공연 `casts.json`을 읽어 파악한다.**

### `schedule.json` (회차 항목, 형식)
```json
{
  "date": "YYYY-MM-DD", "time": "HH:MM", "sid": "...",
  "cast": { "<배역>": ["<배우>"], "<미정배역>": [], "<그룹배역>": ["<팀/조합명>"] },
  "match": { "<대결명>": { "winner": "...", "note": "" } }
}
```
- `[]` = 미정. 채울 때 이 배열에 배우명을 넣는다.
- 팀형 그룹 배역은 **세부 배역이 아니라 팀/조합명** 하나를 넣는다.
- **이미 확정된 배우가 다른 배우로 바뀌는 경우(공지 후 교체·대타 투입 등)에는 배열을 통째로 새 이름으로 바꾸지 않는다.** 원래 배우를 `["이름", 0]`(비중 0) 형태로 남기고 새 배우를 이어 붙인다: `["빌리": [["김우진", 0], "김승주"]]`. 비중 0인 항목은 앱이 "공지됐다 빠진 배우"로 인식해 캐스팅 변경 통계·표시에 쓴다(`js/app.js`의 `parseCastWeighted`). 여러 배우가 회차를 나눠 서는 경우는 `[["이름1", 7], ["이름2", 3]]`처럼 비중 합이 10이 되도록 적는다(기존 사례 참고).

### `casting_by_date.json` (회차 항목, 형식)
```json
"YYYY-MM-DD HH:MM": {
  "<배역>": "<배우>", …,
  "<그룹 세부배역>": "<배우>", …,
  "앙상블": [ ["<배우>", "<역할>"], … ]
}
```
- 키는 **날짜+시간**(`YYYY-MM-DD HH:MM`).
- 팀형 그룹 배역은 **세부 배역을 펼쳐서** 각 배우를 넣는다(팀명 자체는 저장하지 않고 `byTeam` 매핑으로 결정됨).
- 앙상블은 `[이름, 역할]` 배열.

---

## 4. 배역별 배우 판독 & 기존 입력 대조

1. 이미지에서 **`배역 → 배우`**를 모두 뽑는다. 이름은 §2대로 `casts.json`과 대조해 교정.
2. 대상 회차의 **`schedule.json`에 이미 입력된 배우가 있으면 판독값과 비교**한다.
   - **차이가 있으면**: 어느 배역이 어떻게 다른지(입력=X, 보드=Y) **명확히 알린다.**
   - **차이가 없으면**: 배역별로 나열하지 말고 **"일치한다"**고만 말한다.
3. **미정(`[]`)인 배역은 판독값으로 채운다.**

---

## 5. 그룹 캐스트(팀형) 팀 판정 규칙

팀형 그룹 배역은 **팀/조합 단위**로 움직인다. 보드에서 읽은 그룹 배우들을 그 공연 `casts.json`의 각 팀 로스터(`byTeam`)와 대조해 팀을 판정한다.

- **모두 일치**(한 팀의 배우 구성과 판독 배우가 전부 같음) → **그 팀으로 확정.**
- **1명까지 차이** → **더 많은 배우가 속한 팀으로 간주**하고, **그 차이를 알린다.**(예: 원캐 대신 커버/언더 1명 투입)
- **2명 이상 차이** → 마찬가지로 더 많이 속한 팀으로 간주하되, **사용자에게 명시적 확인을 받는다.**

> 팀 판정에는 **팀마다 배우가 갈리는 `byTeam` 세부배역**이 결정적이다. 팀 공통 배역은 팀 구분에 도움이 안 되니 대조에서 배제한다. 어떤 배역이 `byTeam`인지는 그 공연 `casts.json`에서 확인한다.

---

## 6. 앙상블 (명단형 그룹)

- 앙상블은 **`casting_by_date.json`에만 기록**한다(`schedule.json`엔 앙상블 키가 없음).
- 형식: `"앙상블": [ ["배우","역할"], … ]`. 역할 표기는 `casts.json` `members[].roles`를 따른다(예: `앙상블`/`스윙`/`댄스캡틴`).
- **앙상블 구성 변경**(평소와 다른 멤버)이 있으면 **알린다.**
- **앙상블 멤버가 메인 배역을 커버**(대타)로 올라온 경우에도 **알린다.**

---

## 7. 갱신 대상 파일 (둘 다 업데이트)

판독·확정한 캐스팅은 **아래 두 파일을 함께** 갱신한다.

1. **`shows/<id>/schedule.json`** — 대상 회차의 `cast`에서
   - 메인 배역 `[]` → `["배우"]`로 채움.
   - 팀형 그룹 배역 → 판정한 **팀/조합명** 하나.
   - **이미 채워진 배역의 배우가 바뀐 경우**는 §3의 비중 0 규칙대로(`[["원래 배우", 0], "새 배우"]`) 기록한다(완전 교체 금지).
2. **`shows/<id>/casting_by_date.json`** — 같은 회차 키(`YYYY-MM-DD HH:MM`)에
   - 메인 배역 + 그룹 **세부 배역** + **앙상블 배열**까지 확장 기록.
   - 회차 키가 없으면 새로 만든다.

> **대결(match) 결과**는 이 판독 절차에서 건드리지 않는다(보드에 없음). 승부 결과는 별도 요청 시 `schedule.json`의 `match`에 기록.

### 마무리
- 자원(JSON) 외 코드 변경은 없으므로 `requests/NNNN.md`는 만들지 않는다. 공연 **데이터 갱신은 `HISTORY.md`에도 남기지 않는다.**
- 커밋은 항상 먼저 물어본다(프로젝트 규칙).

## 부록. 이미지 판독 실무 팁

- 보드가 작으면 **crop + 확대** 후 읽는다. 예(Pillow):
  ```python
  from PIL import Image
  im = Image.open(path); w,h = im.size
  im.crop((0, int(y0*h), w, int(y1*h))).resize((w*2, int((y1-y0)*h*2))).save('/tmp/row.png')
  ```
- 이름 라벨은 대개 **"배역 ★ 배우"** 형식이 흔하다(공연별로 다를 수 있음). 구분자 뒤가 배우.
- 읽은 이름은 **반드시 `casts.json` 로스터의 이름과 매칭**해 교정(오독 방지).
- 판독 순서·배치(주역 → 그룹 → 앙상블 등)는 공연 보드마다 다르므로, 그 공연 보드에서 실제 배치를 확인하고 필요하면 §2에 기록해 둔다.
