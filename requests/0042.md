## 요청 (작성: Spinouetter)

<!-- 대화로 전달된 요청을 Claude가 옮겨 적음 -->

- 페이지 로딩이 오래 걸리는데, 매번 JSON 데이터 셋을 읽기 때문인지 확인하고 줄일 방법을 찾아 달라.
- (분석 확인 후) 브라우저 동작 중에 JSON을 동적으로 업데이트할 일은 없고 커밋 시점에 확정된다.
  `js?v=`처럼 `.json`에도 버저닝이 가능한가? → 가능하면 그 방식으로.

---

## 정리 (작성: Claude)

120ms 지연(모바일 RTT 수준) 서버로 부팅을 실측한 결과 두 가지 원인:

1. **`cache:"no-store"`** — 모든 JSON을 브라우저 캐시 우회로 받아, 내용이 안 바뀌어도
   매번 풀 다운로드(극장 149KB 포함 ~220KB, 압축 후에도 ~40KB+).
2. **직렬 워터폴** — holidays → matches → resellers를 하나씩 `await`해서 배포 기준
   데이터 요청이 5단계 순차(RTT 200ms면 이것만 ~1초).

### 해결

- **데이터 빌드 버저닝**: 배포 Action이 HTML 마커에 `window.MAKOLLIM_BUILD="<커밋 짧은 SHA>"`를
  주입하고, 앱은 모든 JSON 요청에 `?v=<빌드>`를 붙여 **브라우저 캐시를 활용**(`dataUrl()`).
  데이터는 커밋 시점에 확정 → 배포가 바뀔 때만 URL이 바뀌면 항상 최신. 재방문 시
  변경 없는 파일은 캐시에서 즉시(0 전송).
- **병렬화**: holidays·matches·resellers를 기존 `Promise.all`에 합류 → 데이터 요청
  5단계 → 2단계(공연 정의 → 나머지 전부). 실측(120ms RTT): 데이터 구간 ~970ms → ~370ms.
- **로컬 개발은 그대로**: 빌드 버전 미주입이면 기존 `no-store` 유지(항상 최신).

## 결정 사항 (작성: Claude)

- **버전 값은 커밋 짧은 SHA** — 파일별 해시 대신 배포 단위 버전 하나. 데이터가 커밋 시점에
  확정되므로 충분하고, 매일 cron 재배포(썸네일 갱신)는 같은 SHA라 캐시가 쓸데없이 깨지지 않음.
- **HTML 자체의 캐시(Pages max-age=600)는 그대로 둠** — 배포 직후 최대 10분은 이전 스냅샷을
  볼 수 있으나, 이전 빌드 버전 + 이전 데이터의 **일관된 조합**이라 안전(js?v=와 동일한 특성).
- **finale.js의 fetch는 손대지 않음** — no-store가 아니어서 이미 HTTP 캐시를 타고,
  Finale 탭은 지연 로드라 부팅과 무관. 필요해지면 `dataUrl` 재사용 가능(window에 노출됨).
- resolveShowMeta를 공연 id 선지식으로 더 병렬화(정의 파일과 데이터 동시 시작)하는 것은
  복잡도 대비 이득이 작아 보류 — 버저닝 후 재방문은 캐시 히트라 RTT 자체가 사라짐.
