## 정리 (작성: Claude)

빌리 단일 공연 시절 app.js에 하드코딩된 통계 설정이 멀티 공연(0038) 이후 모든 공연에 새어 들어감:

1. **조합 통계 프리셋** — `getAllComboBlocks()`가 `{roles:["빌리","마이클"]}` 프리셋과 드림 발레 페어를
   comboBlocks에 없으면 무조건 삽입 → 엘리자벳에도 "빌리 × 마이클" 블록이 뜸(삭제 불가).
2. **배역 통계 삭제 금지** — `c.role==="빌리"||c.role==="마이클"` 조건으로 삭제 버튼을 숨김.

### 해결: 공연 정의(shows/<id>.json)의 `stats` 항목으로 이동

```json
"stats": {
  "pinnedRoles": ["빌리", "마이클"],
  "presetCombos": [
    { "id": "billyxmichael", "roles": ["빌리", "마이클"] },
    { "id": "dreamballet", "label": "드림 발레 페어 (빌리 × 성인빌리)",
      "roles": ["빌리", "성인빌리"], "pointRoles": { "빌리": 7 } }
  ]
}
```

- app.js는 `performanceData.statsConfig`로 읽어 `pinnedRoleStats()` / `presetComboEntries()`로 해석.
- `stats`가 없는 공연(엘리자벳)은 프리셋 없음·모든 배역 통계 삭제 가능.
- **드림 발레도 일반화** — `dreamBallet: true` 플래그 대신 `pointRoles`(배역→비중지점)를 가진 프리셋 조합으로.
  `getBillyAtScene7`(씬7=7 하드코딩) → `castAtPoint(entry, point, mode)` 일반 함수,
  `buildDreamBalletHtml`(빌리·성인빌리 하드코딩) → `buildPointComboHtml(def)` 일반 렌더러.
  이로써 **app.js에 배역명 하드코딩 0건**.
- **대결(match)은 이미 전부 JSON** — `shows/<id>/matches.json`의 `{name, point, roles, defaultWinner}` +
  회차 결과는 schedule의 `match` 필드. 코드 수정 없이 README에 문서화만 추가(§7).

## 결정 사항 (작성: Claude)

- **프리셋 블록 id 규칙 유지** — `"preset-" + id` (빌리는 `preset-billyxmichael`·`preset-dreamballet` 그대로).
  기존 사용자 localStorage의 블록 순서·접힘 상태가 id 키로 저장돼 있어 호환 유지.
- **드림 발레는 `pointRoles`로 일반화** — 처음엔 `dreamBallet: true` 플래그로 게이트했으나 후속 요청으로
  "비중 지점 담당 배우 1명으로 좁혀 짝을 세는 조합"이라는 일반 개념(`pointRoles: {배역: 지점}`)으로 승격.
  `matchParticipant`(대결 point)와 같은 개념이라 자연스럽게 맞아떨어짐. 프리셋 렌더는 저장된 항목이 아닌
  **항상 현재 공연 정의로** 그려 옛 저장 형태(`isDreamBallet:true`)도 id로 호환.
- **weighted 모드에서 pointRoles 조합의 카운트는 1회씩**(비중 곱 아님) — 기존 드림 발레 동작 유지.
- **오염 정리는 렌더 시점 필터로** — 엘리자벳 localStorage에 이미 저장된 빌리 프리셋 항목은
  `getAllComboBlocks()`가 현 공연 프리셋 목록에 없는 `isPreset` 항목을 걸러내며 다음 저장 때 사라짐.
  (사용자가 직접 만든 조합은 `isPreset`이 아니므로 건드리지 않음)
