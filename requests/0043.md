# 요청 #0043 — Finale 캐스팅 사진 스프라이트(합친 이미지 + 좌표)로 요청 수 절감

상태: 진행 중 <!-- 대기 / 진행 중 / 완료 / 보류 -->

---

## 요청 (작성: Spinouetter)

- Finale에서 사진을 너무 많이 불러온다(오버레이 열 때 배우 사진 ~58장 개별 요청).
- 캐스팅 보드 사진을 **타일로 하나로 뭉친 다음(스프라이트)** 잘라서 붙이는 게 낫지 않나.
- 사진 비율이 다를 수 있으니 **합친 사진 + 각 이미지의 절대 좌표 저장 파일을 함께 서비스**.
- 커밋 방식(A)으로. 사진 바뀌면 자동 재생성되게.

---

## 정리 (작성: Claude)

### 생성물 (커밋)
- `shows/<id>/images/casting-sprite.jpg` — 배우 사진 전체를 셸프 팩킹한 한 장(타일 최대 400px, q82).
- `shows/<id>/images/casting-sprite.json` — `{version, sprite:"images/casting-sprite.jpg?v=<내용해시>", w, h, tiles:{이름:{x,y,w,h}}}`. 사진 비율이 제각각이어도 각 사진의 **절대 좌표**로 정확히 잘라 쓴다. `sprite`의 `?v=` 는 내용 해시 = 캐시 버스터(바뀔 때만 URL 변경).

### 생성 (sync-cast.py)
- `build_sprite(dst)` 추가 — `shows/<id>/images/*.jpeg`(축소본·플레이스홀더)를 읽어 스프라이트+좌표 생성. `main()`이 사진 축소 후 항상 호출 → **사진 바꾸고 `python3 sync-cast.py [공연id]` 다시 돌리면 스프라이트도 자동 갱신**. temp/cast 원본이 없어도(이미 축소된 사진만 있어도) 재생성 가능.

### 렌더 (finale.js)
- `loadSpriteAtlas()` — 보드 렌더 시 `images/casting-sprite.json`을 1회 로드(no-store).
- `setPhoto` — 스프라이트가 있으면 개별 `<image href>` 대신 스프라이트 1장을 오프셋+클립해 슬롯에 표시. 이름이 스프라이트에 있으면 그 타일(사진·상단정렬), 없으면 플레이스홀더 타일(중앙). → **오버레이 열 때 이미지 요청 58 → 1**.
- 스프라이트가 없는 공연은 기존 개별 사진(+에러 폴백) 그대로. **안전망**: 사진을 바꿨는데 스프라이트를 안 만들면 그 배우만 플레이스홀더(깨지지 않음, sync-cast 재실행으로 해결).
- 내보내기(PNG/JPG/PDF) — `inlinePhotos`가 스프라이트를 1회 로드해 타일별로 슬롯 비율 cover 크롭 → 슬롯 크기 data URI로 교체(+clip 제거·좌표 복원). svg2pdf 호환 구조 유지(개별 크롭 이미지와 동일).

## 결정 사항 (작성: Claude)
- **합친 사진 + 절대 좌표 JSON을 함께 커밋(A)** — 로컬·배포 즉시 동작, CI 폴백 코드 불필요.
- **좌표 = 스프라이트 내 절대 픽셀(x,y,w,h)** — 사진 비율이 달라도 정확히 슬라이스(cover·상단정렬).
- **스프라이트가 사진의 authoritative 소스** — 있는 공연은 개별 요청 0(없는 이름은 바로 플레이스홀더). 개별 사진은 소스·폴백용으로 유지.
- **자동 재생성** — 스프라이트 생성을 sync-cast.py에 합쳐, 사진 갱신 워크플로가 스프라이트까지 최신 유지.
