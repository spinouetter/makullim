# 요청 #0088 — 캐시 버스팅을 커밋 SHA에서 콘텐츠 기반으로 통일

상태: 진행 중 <!-- 대기 / 진행 중 / 완료 / 보류 -->

## 요청 (작성: Spinouetter)

<!-- 대화로 요청됨 (Claude가 정리) -->
- finale.js·stamp.js에도 `?v` 버전을 붙여라.
- 지금 `no-store`가 붙어 있는 게 뭐뭐 있나? `?v=`를 쓰면 `no-store`는 필요 없지 않나 — 정리해라.
- JSON도 커밋 SHA가 아니라 **내용이 바뀌었을 때만** `v`(캐시 버전)를 올려라.

---

## 정리 (작성: Claude)

배포 때마다 커밋 SHA(`MAKULLIM_BUILD`)로 모든 자원의 `?v`가 바뀌어 **내용이 그대로인 파일까지 매번 다시 받는** 문제를 없앤다. 자원별로 "내용이 바뀔 때만" 버스트되도록 캐시 버전을 콘텐츠 기반으로 통일한다.

- **이미지·stamp.json**(요청 0087에서 선행): `imgVer`/`CFG_VER`(소스 콘텐츠 버전)로 전환 완료.
- **stamp.js**: `?v=CFG_VER`가 이미 로컬·배포 모두 붙으므로 `no-store`가 중복 → 제거.
- **finale.js**: 레지스트리·보드 정의(JSON)·배우 사진은 버전이 아예 없었음 → 콘텐츠 버전 `FIN_VER`를 붙인다. 보드 SVG는 이미 자체 `?v`(예: `finale-board.svg?v=28`)가 있으므로 그걸 존중(중복 부착 안 함).
- **app.js 데이터 JSON**(schedule·casts·grades·matches·resellers·casting_by_date·theatre·holidays): 이 파일들은 협업자가 자주 커밋해 바꾸므로 JS에 상수를 손으로 유지하기 어렵다. 그래서 **빌드 시 콘텐츠 해시**로 처리한다 — 배포(`pages.yml`)가 각 JSON의 sha1 앞 10자를 모아 `window.MAKULLIM_ASSET_VER` 맵으로 스텁에 주입하고, `dataUrl()`이 그 해시를 `?v=`로 붙인다(맵에 없으면 빌드 SHA로 폴백, 로컬은 `no-store`).

`no-store` 정리 결과: `?v` 콘텐츠 버전이 붙는 stamp.json·이미지에서는 제거. app.js 데이터 JSON의 로컬 개발용 `no-store`는 **로컬엔 버전 주입이 없으므로 유지**(배포본은 `?v` 해시라 캐시를 그대로 씀).

## 할 일 (작성: Claude)

- [x] stamp.js: `no-store` 헬퍼 제거, `?v=CFG_VER`만 사용
- [x] finale.js: `FIN_VER` + `verUrl()` 도입, 레지스트리·보드 정의·사진에 부착(SVG 자체 `?v`는 존중)
- [x] app.js: `dataUrl()`이 `MAKULLIM_ASSET_VER` 콘텐츠 해시 맵을 1순위로 사용, 빌드 SHA는 폴백
- [x] pages.yml: 데이터 JSON 콘텐츠 해시 맵 생성 후 슬러그 스텁·루트에 주입(`.github/workflows/README.md` 갱신)
- [x] 버전 쿼리 올림: index.html의 `app.js?v=209`·`finale.js?v=55`·`stamp.js?v=24`
- [x] 로컬 검증(Playwright): 스탬프/피날레 렌더, 자원 200, `dataUrl`이 해시/폴백을 올바르게 선택

## 결정 사항 (작성: Claude)

- **왜 app.js 데이터만 빌드 해시인가**: schedule·casting_by_date 등은 데이터 커밋으로 자주 바뀌어 사람이 JS 상수(`FIN_VER`류)를 매번 올리기 비현실적 → 빌드가 자동으로 내용 해시. finale/stamp 자원은 상대적으로 드물게 바뀌므로 소스 상수로 충분.
- **SVG 이중 버전 방지**: `verUrl()`은 경로에 이미 `?`가 있으면 그대로 둔다. 보드 정의의 `background.src`가 자체 `?v`를 들고 있어 그것이 그 파일의 권위 있는 콘텐츠 버전.
- **폴백 유지**: `MAKULLIM_ASSET_VER`에 없는 경로는 빌드 SHA로 폴백해, 맵 생성 누락이 있어도 캐시가 stale해지지 않음(과다 버스트일 뿐 안전).
- **워크플로 변경**(pages.yml)은 규칙에 따라 requests가 아니라 `.github/workflows/README.md` + git history로 관리 — 이 문서에는 앱(app.js) 쪽 계약만 요약.
